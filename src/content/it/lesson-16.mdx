## La Struct Professionale: Flexible Array Members

Dimentichiamo i cast manuali e i salti di 4 o 8 byte. C ci offre uno strumento elegante introdotto con lo standard C99: i **Flexible Array Members**.

```c
struct pls {
    uint32_t len;        // 4 byte: Lunghezza
    uint32_t ref_count;  // 4 byte: Contatore referenze
    uint32_t magic;      // 4 byte: Numero magico per sicurezza
    char str[];          // Flexible Array: punta all'offset 12!
};
```

<Callout type="pro" title="La Fisica del Flexible Array">
  Perché `char str[]` e non `char *str`? 1. **Unica Allocazione**: Con `*str`
  dovresti fare due `malloc` (una per la struct, una per il testo). Con il
  Flexible Array ne fai **una sola**, includendo lo spazio extra. 2. **Cache
  Locality**: Poiché header e testo sono fisicamente vicini, la CPU carica tutto
  in un'unica Cache Line. Questo abbatte i tempi di accesso rispetto a un
  puntatore che punta altrove nella RAM. 3. **Sizeof**: Nota che `sizeof(struct
  pls)` restituirà solo 12 byte. L'array `str[]` non occupa spazio "nel tipo", è
  solo un'etichetta per l'indirizzo che segue l'ultimo campo.
</Callout>

## Implementazione: La libreria `pls`

Ecco il cuore della gestione dinamica discussa da Salvatore.

<CodeEditor 
  initialCode={`#include <stdio.h>
#include <stdlib.h>
#include <stdint.h>
#include <string.h>

#define PLS_MAGIC 0xDEADBEEF

struct pls {
uint32_t len;
uint32_t ref_count;
uint32_t magic;
char str[];
};

// Crea una stringa con ref_count = 1
struct pls* ps_create(const char *init) {
uint32_t lens = strlen(init);
// Allociamo Header + Testo + Null Terminator in un colpo solo
struct pls \*p = malloc(sizeof(struct pls) + lens + 1);
if (!p) return NULL;
p->len = lens;
p->ref_count = 1;
p->magic = PLS_MAGIC;
strcpy(p->str, init);
return p;
}

// Incrementa la fiducia nell'oggetto
void ps_retain(struct pls \*p) {
if (p->magic != PLS_MAGIC) {
fprintf(stderr, "Errore: Oggetto corrotto o già liberato!\\n");
abort();
}
p->ref_count++;
}

// Decrementa e libera se necessario
void ps_release(struct pls \*p) {
if (p->magic != PLS_MAGIC) abort();
p->ref_count--;
if (p->ref_count == 0) {
// "Sabotaggio" di sicurezza: invalidiamo il magic prima della free
p->magic = 0x00000000;
free(p);
printf("Oggetto distrutto fisicamente.\\n");
}
}

int main() {
struct pls \*s1 = ps_create("esperto");

    // Simulo una condivisione
    struct pls *s2 = s1;
    ps_retain(s2); // Ora ref_count è 2

    printf("Stringa: %s (Refs: %d)\\n", s1->str, s1->ref_count);

    ps_release(s1); // ref_count torna a 1
    ps_release(s2); // ref_count va a 0 -> FREE!

    return 0;

}`}
language="c"
/>

<Callout type="pro" title="Magic Numbers e Debugging Visuale">
  Perché `0xDEADBEEF`? È uno dei "Magic Words" più famosi perché è facile da
  individuare in un **Hexdump** manuale della RAM. Se vedi `ef be ad de` (in
  Little Endian) nel debugger, sai esattamente dove inizia l'header della tua
  stringa. Se vedi zeri, l'oggetto è stato sabotato dalla `ps_release`.
</Callout>

<Callout type="danger" title="Atomic Reference Counting">
  Salvatore avverte: in un programma multi-thread, `ref_count++` non è sicuro!
  Due thread potrebbero incrementarlo contemporaneamente portando a corruzione
  della memoria. Nei sistemi professionali si usano istruzioni CPU atomiche come
  `lock xadd` su x86.
</Callout>
