## La Struct Professionale

Dimentichiamo i cast manuali e i salti di 4 o 8 byte. C ci offre uno strumento elegante: i **Flexible Array Members**.

```c
struct pls {
    uint32_t len;        // 4 byte: Lunghezza
    uint32_t ref_count;  // 4 byte: Contatore referenze
    uint32_t magic;      // 4 byte: Numero magico per sicurezza
    char str[];          // Flexible Array: punta all'offset 12!
};
```

### Layout di Memoria "Compatto"

Grazie alla `char str[]`, l'header e il testo vivono in un **unico blocco contiguo**. Questo è fondamentale per la cache della CPU e per la semplicità di allocazione.

## Implementazione: La libreria `pls`

Ecco il cuore della gestione dinamica discussa da Salvatore.

<CodeEditor 
  initialCode={`#include <stdio.h>
#include <stdlib.h>
#include <stdint.h>
#include <string.h>

#define PLS_MAGIC 0xDEADBEEF

struct pls {
    uint32_t len;
    uint32_t ref_count;
    uint32_t magic;
    char str[];
};

// Crea una stringa con ref_count = 1
struct pls* ps_create(const char *init) {
    uint16_t lens = strlen(init);
    struct pls *p = malloc(sizeof(struct pls) + lens + 1);
    p->len = lens;
    p->ref_count = 1;
    p->magic = PLS_MAGIC;
    strcpy(p->str, init);
    return p;
}

// Incrementa la fiducia nell'oggetto
void ps_retain(struct pls *p) {
    if (p->magic != PLS_MAGIC) abort(); // Difesa!
    p->ref_count++;
}

// Decrementa e libera se necessario
void ps_release(struct pls *p) {
    if (p->magic != PLS_MAGIC) abort();
    p->ref_count--;
    if (p->ref_count == 0) {
        p->magic = 0; // Invalido prima di liberare
        free(p);
        printf("Oggetto distrutto fisicamente.\\n");
    }
}

int main() {
    struct pls *s1 = ps_create("Hacker");
    
    // Simulo una condivisione
    struct pls *s2 = s1; 
    ps_retain(s2); // Ora ref_count è 2
    
    printf("Stringa: %s (Refs: %d)\\n", s1->str, s1->ref_count);
    
    ps_release(s1); // ref_count torna a 1
    ps_release(s2); // ref_count va a 0 -> FREE!
    
    return 0;
}`} 
  language="c" 
/>

Salvatore implementa un Reference Counting manuale. Ogni volta che un pezzo di codice "prende" l'oggetto chiama `retain`, quando finisce chiama `release`.

<Callout type="danger" title="Reference Counting Atomico">
  Salvatore avverte: in un programma multi-thread, `ref_count++` non è sicuro! Due thread potrebbero incrementarlo contemporaneamente portando a corruzione della memoria. Nei sistemi professionali (come il kernel Linux), si usano funzioni "atomiche" speciali che garantiscono la sicurezza tra i thread.
</Callout>

<Callout type="danger" title="Integrità dei Magic Numbers">
  Il valore `0xDEADBEEF` non è solo per bellezza. Se durante un `ps_release` il Magic Number è diverso, significa che qualcuno ha scritto fuori dai bordi (Buffer Overflow) corrompendo l'header. Fermare il programma immediatamente (`abort()`) è meglio che lasciarlo girare con dati corrotti.
</Callout>

<Callout type="hacker" title="Magic Numbers">
Perché Salvatore usa `0xDEADBEEF`? È un valore facile da riconoscere in un esadecimale (hexdump). Se vedi `00000000` dove dovrebbe esserci il magic, sai che stai usando un oggetto già liberato!
</Callout>

<Callout type="info" title="Debugging con le Macro">
Salvatore suggerisce di usare le macro del preprocessore (`#ifdef DEBUG`) per includere il campo `magic` solo durante lo sviluppo. In questo modo, la versione "finale" del software sarà più leggera di 4 byte per ogni stringa, ma avrai avuto tutta la sicurezza necessaria durante i test.
</Callout>

### Manutenzione delle Referenze
È fondamentale che ogni `ps_retain` sia accoppiata a una `ps_release`. Se dimentichi una release, hai un **Memory Leak**. Se fai una release di troppo, hai un **Use-After-Free** o un crash.
