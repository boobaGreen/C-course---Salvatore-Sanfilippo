## Cicli: for vs while
Come spiega Salvatore, un ciclo `for` è spesso solo un modo più compatto di scrivere un `while`. 

```c
// Versione WHILE
int i = 0;
while (i < 5) {
    // codice
    i++;
}

// Versione FOR (più elegante)
for (int i = 0; i < 5; i++) {
    // codice
}
```

Usiamo il `for` quando sappiamo già quante volte iterare, e il `while` quando la condizione dipende da qualcosa che accadrà nel codice.

<Callout type="hacker" title="For: Flessibilità Totale">
In C, le tre parti del `for` (inizializzazione, condizione, incremento) sono opzionali!
- `for(;;)` è un modo idiomatico per scrivere un **loop infinito**.
- Puoi omettere l'incremento se lo fai dentro il corpo del ciclo.
</Callout>

### Esempio Pratico: Ricerca in un Array
Salvatore ci mostra come usare il `break` per fermare un ciclo non appena troviamo quello che cerchiamo.

```c
int a[] = {1, 5, 8, 4, 0}; // 0 è il terminatore
int i;
for (i = 0; a[i] != 0; i++) {
    if (a[i] == 8) break; // Trovato! Esci subito.
}

if (a[i] == 0) printf("Non trovato\n");
else printf("Trovato in posizione %d\n", i);
```

<Callout type="warning" title="Scope della Variabile i">
Se dichiari `for (int i = 0; ...)` dentro il ciclo, la variabile `i` sparisce appena il ciclo finisce. Se ti serve sapere a che punto si è fermato il ciclo (come nell'esempio sopra), devi dichiarare la `i` **fuori** dal `for`.
</Callout>

## La Ricorsione e lo Stack

Quando una funzione chiama se stessa, il computer non ricomincia da zero. "Congela" la funzione attuale e ne apre una nuova sopra di essa (un nuovo "frame").

<RecursionVisualizer />

<Callout type="hacker" title="Tail Call Optimization (TCO)">
Se l'ultima cosa che una funzione fa è chiamare se stessa, un buon compilatore può ottimizzarla trasformandola in un salto (`goto`), risparmiando memoria. Se non ti fidi del compilatore, puoi farlo a mano:
```c
loop:
    // ... fai qualcosa ...
    start = start + 1;
    goto loop; // Ricorsione manuale!
```
</Callout>

<Callout type="warning" title="Stack Overflow">
Se la ricorsione non ha una 'condizione di uscita' (base case), continuerà a creare nuovi frame fino a esaurire la memoria dedicata allo stack. Questo causa il famoso crash 'Stack Overflow'.
</Callout>

<SwitchBoard />

<Callout type="hacker" title="Jump Tables">
Sotto il cofano, se i casi sono vicini (es. 1, 2, 3), il compilatore non fa tanti `if`. Crea una **Jump Table**: una tabella di indirizzi dove il processore "salta" istantaneamente al caso giusto. È incredibilmente veloce!
</Callout>

<Callout type="info" title="L'importanza del break">
Nello `switch`, se dimentichi il `break;`, il codice continuerà a eseguire anche i casi successivi! Questo comportamento si chiama 'fall-through'.
</Callout>

## Codice Reale: Il Glob Matching di Redis
Salvatore ci mostra come usa questi concetti nel codice vero. La sua funzione per il matching delle stringhe (es. `*`, `?`, `[a-z]`) usa sia lo `switch` che la ricorsione per gestire i vari caratteri del pattern.

```c
// Semplificazione del glob matching di Salvatore
int glob_match(const char *pattern, const char *string) {
    switch (pattern[0]) {
        case '*': // Ricorsione per gestire l'asterisco
            // ... logica complessa ...
            break;
        case '?': // Salta un carattere
            // ...
            break;
        default: // Match letterale
            if (pattern[0] != string[0]) return 0;
    }
}
```

<KeyConcepts title="Hacker Key Concepts">
  - **for = while**: Il ciclo `for` è solo un modo più compatto per scrivere un `while` con inizializzazione e incremento.
  - **Stack vs Ricorsione**: Ogni chiamata ricorsiva occupa spazio sullo Stack (frame). Attenzione a non esaurirlo!
  - **Jump Tables**: Lo `switch` è velocissimo perché usa tabelle di salto invece di una lunga serie di comparazioni individuali.
</KeyConcepts>
