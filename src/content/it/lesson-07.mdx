## Cicli: for vs while

Come spiega Salvatore, un ciclo `for` è spesso solo un modo più compatto di scrivere un `while`.

```c
// Versione WHILE
int i = 0;
while (i < 5) {
    // codice
    i++;
}

// Versione FOR (più elegante)
for (int i = 0; i < 5; i++) {
    // codice
}
```

<Callout type="info" title="Zucchero Sintattico">
  Il ciclo `for` è quello che i programmatori chiamano "Zucchero Sintattico":
  non aggiunge nuove funzionalità al linguaggio, ma rende solo più dolce
  (leggibile e compatta) una struttura che potevi già scrivere con il `while`.
</Callout>

### Trasformazione: Da For a While

Per capire davvero il `for`, impara a smontarlo:

1. La prima parte (inizializzazione) va **prima** del ciclo.
2. La seconda parte (condizione) è il **motore** del `while`.
3. La terza parte (incremento) va alla **fine** del blocco di codice.

Usiamo il `for` quando sappiamo già quante volte iterare, e il `while` quando la condizione dipende da qualcosa che accadrà nel codice.

<Callout type="pro" title="For: Flessibilità Totale">
In C, le tre parti del `for` (inizializzazione, condizione, incremento) sono opzionali!
- `for(;;)` è un modo idiomatico per scrivere un **loop infinito**.
- Puoi omettere l'incremento se lo fai dentro il corpo del ciclo.

<Callout type="warning" title="Loop Unrolling (Srotolamento)">
  Il compilatore può decidere di "srotolare" un ciclo `for` corto (es. 4 iterazioni) scrivendo direttamente le 4 istruzioni una dopo l'altra. Questo elimina il costo del controllo della condizione `i < 4` a ogni giro e aiuta la **Pipeline** della CPU a fluire senza interruzioni dovute ai salti.
</Callout>
</Callout>

### Esempio Pratico: Ricerca in un Array

Salvatore ci mostra come usare il `break` per fermare un ciclo non appena troviamo quello che cerchiamo.

```c
int a[] = {1, 5, 8, 4, 0}; // 0 è il terminatore
int i;
for (i = 0; a[i] != 0; i++) {
    if (a[i] == 8) break; // Trovato! Esci subito.
}

if (a[i] == 0) printf("Non trovato\n");
else printf("Trovato in posizione %d\n", i);
```

<Callout type="warning" title="Scope della Variabile i">
  Se dichiari `for (int i = 0; ...)` dentro il ciclo, la variabile `i` sparisce
  appena il ciclo finisce. Se ti serve sapere a che punto si è fermato il ciclo
  (come nel caso della ricerca dell'8 sopra), devi dichiarare la `i` **fuori**
  dal `for`.
</Callout>

<Callout type="danger" title="Il Problema del 'Dangling Else'">
  Quando annidi degli `if` senza usare le graffe, l' `else` si lega sempre all'
  `if` più vicino. È una delle fonti di bug più bastarde del C. **Consiglio di
  Salvatore:** usa sempre le graffe `{}`, anche per una sola riga!
</Callout>

## La Ricorsione e lo Stack

Quando una funzione chiama se stessa, il computer non ricomincia da zero. "Congela" la funzione attuale e ne apre una nuova sopra di essa (un nuovo "frame").

<RecursionVisualizer />

<Callout type="pro" title="Tail Call Optimization (TCO)">
Se l'ultima cosa che una funzione fa è chiamare se stessa, un buon compilatore può ottimizzarla trasformandola in un salto (`goto`), risparmiando memoria.

**Attenzione**: la TCO funziona solo se non c'è "lavoro residuo".

- `return f(x);` -> Ottimizzabile.
- `return 1 + f(x);` -> **NON** ottimizzabile (la CPU deve ricordarsi di aggiungere 1 dopo che la funzione torna).

Salvatore ci ha mostrato come farlo a mano per evitare di consumare lo stack:

```c
loop:
    if (start > end) return;
    printf("%d ", start);
    start = start + 1;
    goto loop; // Ricorsione manuale (iterazione)!
```

In questo modo l'indirizzo di memoria di `start` rimarrà sempre lo stesso, perché non creiamo nuovi frame.

</Callout>

<Callout type="warning" title="Stack Overflow">
  Se la ricorsione non ha una 'condizione di uscita' (base case), continuerà a
  creare nuovi frame fino a esaurire la memoria dedicata allo stack. Questo
  causa il famoso crash 'Stack Overflow'.
</Callout>

<SwitchBoard />

<Callout type="pro" title="Jump Tables">
Sotto il cofano, se i casi sono vicini (es. 1, 2, 3), il compilatore non fa tanti `if`. Crea una **Jump Table**: un array di indirizzi di memoria. Il processore prende il valore, lo usa come indice della tabella e salta direttamente al codice giusto. È incredibilmente veloce!

**Il Limite**: Se i casi sono molto distanti (es. `case 1` e `case 1000000`), il compilatore non può creare una tabella così grande. In quel caso "torna indietro" e usa una serie di `if` o una ricerca binaria.

</Callout>

<Callout type="info" title="La Velocità dello Switch">
  A differenza di una catena di `if-else` che è **O(N)** (più casi hai, più
  lento diventa), lo `switch` con Jump Table è **O(1)**. Non importa se hai 10 o
  100 casi: il tempo per saltare a quello giusto rimane lo stesso.
</Callout>

<Callout type="info" title="L'importanza del break">
  Nello `switch`, se dimentichi il `break;`, il codice continuerà a eseguire
  anche i casi successivi! Questo comportamento si chiama 'fall-through'.
</Callout>

## Codice Reale: Il Glob Matching di Redis

Salvatore ci mostra come usa questi concetti nel codice vero. La sua funzione per il matching delle stringhe (es. `*`, `?`, `[a-z]`) usa sia lo `switch` che la ricorsione per gestire i vari caratteri del pattern.

```c
// Semplificazione del glob matching di Salvatore
int glob_match(const char *pattern, const char *string) {
    switch (pattern[0]) {
        case '*': // Ricorsione per gestire l'asterisco
            // ... logica complessa ...
            break;
        case '?': // Salta un carattere
            // ...
            break;
        default: // Match letterale
            if (pattern[0] != string[0]) return 0;
    }
}
```
