## L'Architettura dell'Oggetto: TFObject

Per rappresentare un linguaggio, abbiamo bisogno di un tipo di dato universale. In ToyFort usiamo una `struct` che racchiude una `union`.

<Callout type="pro" title="Risparmio Hardware">
  Usando la `union` all'interno di `TFObject`, garantiamo che ogni oggetto
  occupi solo la memoria del tipo più grande (solitamente la lista o la
  stringa), indipendentemente dal fatto che contenga solo un piccolo intero. È
  l'ottimizzazione che permette a linguaggi come Python o Ruby di gestire
  milioni di oggetti.
</Callout>

### Anatomia di una Lista Eterogenea

Il campo `TFObject **ele` è fondamentale: è un array di puntatori. Perché non un array di oggetti?

1. **Eterogeneità**: Ogni cella può puntare a un oggetto di tipo diverso.
2. **Condivisione**: Più liste possono puntare allo stesso oggetto (grazie al `ref_count`).
3. **Ricorsività**: Una lista può contenere un'altra lista, creando alberi di esecuzione.

## Logica Stack-based: RPN (Reverse Polish Notation)

Dimentica le parentesi. In ToyFort scrivi `5 5 +`.

1. La CPU (tramite il nostro interprete) mette 5 nello stack.
2. Mette un altro 5.
3. Quando incontra `+`, preleva gli ultimi due, li somma e rimette il risultato.

## Implementazione: xmalloc e Safety

#include <stdlib.h>

// Wrapper di sicurezza: se la memoria finisce, il programma si ferma invece di crashare male
void *xmalloc(size_t size) {
void *p = malloc(size);
if (!p) {
fprintf(stderr, "FATAL: Out of memory\\n");
exit(1);
}
return p;
}

typedef enum { TF_INT, TF_STR, TF_LIST } TFType;

struct TFObject {
TFType type;
int ref_count;
union {
long long i;
char \*s;
struct {
struct TFObject \*\*ele;
int len;
} list;
};
};

int main() {
printf("TFObject pronto per l'allocazione!\\n");
return 0;
}`}
