## Il "Hello World" in C

Il C è un linguaggio molto vecchio che si è evoluto relativamente poco nella sintassi base, ma moltissimo nei **compilatori**. Rimane un linguaggio fatto di "poche idee", che vanno però assimilate a fondo. Essendo un **linguaggio compilato**, non possiamo eseguire direttamente il codice sorgente: dobbiamo prima tradurlo in linguaggio macchina.

Ecco il classico programma di partenza, in sole 5 righe:

```c
#include <stdio.h>

int main() {
    printf("Hello, World!\n");
    return 0;
}
```

<Callout type="tip" title="Consiglio di Salvatore">
  Non cercate di capire tutto subito. In C, molti concetti si chiariscono solo
  quando scendiamo di un livello e vediamo cosa succede nella memoria o nella
  CPU.
</Callout>

### Come si compila ed esegue?

C è un linguaggio **compilato**. Questo significa che il codice che scriviamo non viene letto riga per riga da un interprete (come Python o JS), ma trasformato integralmente in un file binario prima di poter essere lanciato.

<Infographic
  src="/images/c_compilation_pipeline_v2.png"
  alt="La Pipeline di Compilazione (Dettagliata)"
  caption="Le 4 fasi: Preprocessing, Compilazione, Assembly e Linking."
/>

Dobbiamo usare il **compilatore C** (come `cc`, `gcc` o `clang`):

```bash
cc hello.c
```

### La Democratizzazione del C

Storicamente, i compilatori erano strumenti costosi e proprietari (come l'**Intel C Compiler - ICC**). La svolta è arrivata con **Richard Stallman** e il progetto **GCC** (GNU Compiler Collection). Regalando un compilatore potente e gratuito al mondo, Stallman ha permesso a chiunque di imparare e usare il C, cambiando per sempre la storia del software libero.

Questo comando genera un nuovo file eseguibile (tipicamente chiamato `a.out` su sistemi Unix). Questo file è un **binario** contenente il linguaggio macchina e alcune intestazioni specifiche del sistema operativo (Linux, macOS, Windows). Possiamo eseguirlo digitando:

## Dietro le quinte: l'Assembly e le Ottimizzazioni

Possiamo chiedere al compilatore di fermarsi prima di generare il binario finale, facendoci mostrare il codice **Assembly** intermedio, usando il flag `-S`. Inoltre, aggiungendo il flag `-O2`, gli chiediamo di applicare delle ottimizzazioni avanzate.

Questo flag è fondamentale perché ci permette di ispezionare esattamente come il compilatore ha tradotto il nostro codice C, senza dover "de-compilare" un eseguibile binario (che è molto più complesso).

Esempio di utilizzo:

```bash
cc -O2 -S hello.c
```

Questo comando non crea un `a.out`, ma genera un file testuale chiamato `hello.s` contenente le istruzioni Assembly.

L'Assembly è il linguaggio "quasi" macchina: è leggibile dagli esseri umani ma corrisponde 1:1 alle istruzioni che la CPU eseguirà.

<AssemblyExplorer />

<Callout type="hacker" title="Hacker Mindset">
  Guardare l'Assembly non è un esercizio accademico. È l'unico modo per capire
  *davvero* quanto è efficiente il tuo codice e cosa sta facendo il compilatore
  alle tue spalle.
</Callout>

Il compilatore, grazie al livello di ottimizzazione `-O2`, accorgendosi che stiamo stampando una stringa semplice seguita da un ritorno a capo (`\n`), sostituisce automaticamente la `printf` con la funzione `puts`, che è più veloce ed efficiente per questo scopo specifico.

<Callout type="info" title="Intelligenza del Compilatore">
  Questo è il primo esempio di **ottimizzazione trasparente**. Il compilatore
  non si limita a tradurre, ma "capisce" l'intento del codice e lo riscrive per
  essere più performante. È per questo che il C, nonostante l'età, rimane
  imbattibile: i compilatori moderni hanno accumulato 50 anni di astuzie
  matematiche.
</Callout>

## Facciamo a pezzi il programma

Analizziamo ogni riga per capire cosa succede realmente:

### 1. Il Preprocessore e `#include`

```c
#include <stdio.h>
```

Questa riga non fa parte del linguaggio C vero e proprio, ma è una **direttiva per il preprocessore** (tutte le direttive iniziano con il `#`). Prima della compilazione vera e propria, il preprocessore prende il contenuto del file `stdio.h` (un file di libreria standard) e lo "scompatta" direttamente all'interno del nostro file.

<Callout type="info">
  Il C è un linguaggio "nudo". Senza librerie, non sa nemmeno come stampare un
  carattere a video. Tutto ciò che è "utile" si trova nella Libreria Standard.
</Callout>

<Callout type="warning" title="Sicurezza: printf vs puts">
  Non usare mai `printf(buffer)` se `buffer` proviene da un input utente! Se
  l'utente inserisce stringhe tipo `%s%s%s`, può causare un **Format String
  Attack**. `puts` è intrinsecamente più sicuro perché non interpreta i
  segnaposto.
</Callout>

### 2. File Header (`.h`) e Prototipi di Funzione

In un file header (come `stdio.h`) non c'è il codice che _esegue_ la `printf`, ma ci sono i **prototipi di funzione**.
