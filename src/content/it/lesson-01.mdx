## Il "Hello World" in C

Il C è un linguaggio molto vecchio che si è evoluto relativamente poco nella sintassi base, ma moltissimo nei **compilatori**. Rimane un linguaggio fatto di "poche idee", che vanno però assimilate a fondo. Essendo un **linguaggio compilato**, non possiamo eseguire direttamente il codice sorgente: dobbiamo prima tradurlo in linguaggio macchina.

Ecco il classico programma di partenza, in sole 5 righe:

```c
#include <stdio.h>

int main() {
    printf("Hello, World!\n");
    return 0;
}
```

<Callout type="tip" title="Consiglio di Salvatore">
  Non cercate di capire tutto subito. In C, molti concetti si chiariscono solo quando scendiamo di un livello e vediamo cosa succede nella memoria o nella CPU.
</Callout>

### Come si compila ed esegue?

C è un linguaggio **compilato**. Questo significa che il codice che scriviamo non viene letto riga per riga da un interprete (come Python o JS), ma trasformato integralmente in un file binario prima di poter essere lanciato.

Dobbiamo usare il **compilatore C** (come `cc`, `gcc` o `clang`):

Questo comando genera un nuovo file eseguibile (tipicamente chiamato `a.out` su sistemi Unix). Questo file è un **binario** contenente il linguaggio macchina e alcune intestazioni specifiche del sistema operativo (Linux, macOS, Windows). Possiamo eseguirlo digitando:

## Dietro le quinte: l'Assembly e le Ottimizzazioni

Possiamo chiedere al compilatore di fermarsi prima di generare il binario finale, facendoci mostrare il codice **Assembly** intermedio, usando il flag `-S`.

L'Assembly è il linguaggio "quasi" macchina: è leggibile dagli esseri umani ma corrisponde 1:1 alle istruzioni che la CPU eseguirà.

<AssemblyExplorer />

<Callout type="hacker" title="Hacker Mindset">
  Guardare l'Assembly non è un esercizio accademico. È l'unico modo per capire *davvero* quanto è efficiente il tuo codice e cosa sta facendo il compilatore alle tue spalle.
</Callout>

Come hai visto nell'explorer qui sopra, se compiliamo con il flag `-O2` (ottimizzazione di livello 2), noteremo una cosa sorprendente: **la funzione `printf` scompare**! 

Il compilatore, accorgendosi che stiamo stampando una stringa semplice seguita da un ritorno a capo (`\n`), sostituisce automaticamente la `printf` con la funzione `puts`, che è più veloce ed efficiente per questo scopo specifico.

## Facciamo a pezzi il programma

Analizziamo ogni riga per capire cosa succede realmente:

### 1. Il Preprocessore e `#include`
```c
#include <stdio.h>
```
Questa riga non fa parte del linguaggio C vero e proprio, ma è una **direttiva per il preprocessore** (tutte le direttive iniziano con il `#`). Prima della compilazione vera e propria, il preprocessore prende il contenuto del file `stdio.h` (un file di libreria standard) e lo "scompatta" direttamente all'interno del nostro file. 

<Callout type="info">
  Il C è un linguaggio "nudo". Senza librerie, non sa nemmeno come stampare un carattere a video. Tutto ciò che è "utile" si trova nella Libreria Standard.
</Callout>

### 2. File Header (`.h`) e Prototipi di Funzione

In un file header (come `stdio.h`) non c'è il codice che *esegue* la `printf`, ma ci sono i **prototipi di funzione**.

<ComparisonTable 
  title="Prototipo vs Definizione"

Il prototipo serve a dire al compilatore: *"Esiste una funzione chiamata `printf` che accetta questi parametri e restituisce un numero intero. Fidati, la troverai più tardi"*.

<KeyConcepts title="Ricapitolando">
  - Il C non è eseguito, ma **compilato**.
  - Il compilatore sa ottimizzare il codice in modo intelligente (es. trasformando `printf` in `puts`).
  - Il **preprocessore** prepara il file prima della compilazione.
  - I file **Header (`.h`)** contengono i **prototipi** delle funzioni.
</KeyConcepts>
