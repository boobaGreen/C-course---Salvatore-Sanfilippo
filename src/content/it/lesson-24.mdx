## Lo Stack alla Prova del C

In ToyFort, ogni funzione C riceve il `TFContext`. Questa struttura contiene la "Pila" dei dati. Quando scrivi `+`, la funzione C corrispondente deve:

1. Verificare che ci siano almeno 2 elementi.
2. Estrarre (POP) i valori.
3. Eseguire l'operazione.
4. Inserire (PUSH) il risultato.

<Callout type="pro" title="Gestione della Proprietà (Ownership)">
  Ogni `pop` dallo stack ti dà la proprietà dell'oggetto. Se dimentichi di
  chiamare `tf_object_release()`, hai creato un **Memory Leak**. Se lo chiami
  due volte, hai un **Double Free**. La disciplina è l'unica difesa del
  programmatore C.
</Callout>

## Amortized O(1): Il Segreto delle Liste Veloci

Perché Salvatore non alloca esattamente lo spazio necessario per ogni nuovo elemento?
Se dovessimo chiamare `realloc()` per ogni singolo elemento, perderemmo tempo a copiare i dati ogni volta. Invece, raddoppiamo la capacità ogni volta che la riempiamo.

## Implementazione: Il cuore dell'Eval

<CodeEditor 
  initialCode={`#include <stdio.h>
#include <string.h>

void tf_eval(TFContext *ctx, TFObject *obj) {
if (obj->type == TF_INT || obj->type == TF_STR || obj->type == TF_LIST) {
// Valore letterale: mettilo nello stack
tf_stack_push(ctx, tf_object_retain(obj));
} else if (obj->type == TF_SYM) {
// Simbolo: cercalo nel dizionario
TFFunc f = tf_dict_lookup(ctx, obj->s);
if (f) {
f(ctx); // Esegui la funzione C associata
} else {
fprintf(stderr, "Errore: Simbolo '%s' sconosciuto\\n", obj->s);
}
}
}
`}
language="c"
/>

<Callout type="warning" title="Branch Prediction">
  Il ciclo `eval` è una sfida per la CPU. Poiché le funzioni chiamate cambiano
  continuamente, il **Branch Predictor** fatica a prevedere dove salterà il
  codice. Questo "overhead di dispacciamento" è il prezzo da pagare per la
  flessibilità di un interprete rispetto al codice compilato.
</Callout>
