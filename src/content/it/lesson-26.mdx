## Il Ciclo di Esecuzione: `execute_program`

L'interprete ToyFort cicla su una lista. Se trova un valore, lo "puscia" sullo stack; se trova un simbolo, lo esegue.

<Callout type="hacker" title="Perché xrealloc?">
Salvatore usa `xrealloc` invece di `realloc` perché, man mano che l'interprete cresce, le riallocazioni diventano frequenti. Se `realloc` fallisce a metà di una struttura complessa l'unica cosa sicura da fare è un "Panic" (chiusura immediata) invece di tentare di gestire un errore impossibile.
</Callout>

## Gestione Memoria: Retain e Release

In C, gestire la memoria di un interprete è complesso perché gli oggetti possono essere condivisi. Salvatore implementa un sistema di **Reference Counting**.

<ComparisonTable 
  title="Gestione Manuale vs Ref Counting"
  headers={["Metodo", "Vantaggio", "Svantaggio"]}
  rows={[
    ["Malloc/Free classica", "Massima velocità", "Rischio enorme di double-free o leak"],
    ["Reference Counting", "Gestione semi-automatica", "Overhead di memoria e rischio di cicli"],
    ["Garbage Collector", "Completamente automatico", "Pause 'Stop the world' e complessità"]
  ]}
/>

### Come funziona nel codice:

```c
void retain_object(TFObject *o) {
    o->ref_count++;
}

void release_object(TFObject *o) {
    o->ref_count--;
    if (o->ref_count == 0) {
        free_object(o); // Libera davvero la memoria
    }
}
```

<Callout type="warning" title="La trappola del Pop">
Quando estrai un oggetto dallo stack con una `pop`, ne diventi il proprietario unico. Se ti dimentichi di chiamare `release` alla fine della funzione, quell'oggetto non verrà mai liberato!
</Callout>
