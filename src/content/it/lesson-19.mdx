## File Descriptors: 0, 1, 2

Ogni processo Linux nasce con tre canali di comunicazione già aperti (ereditati dal processo padre):

- **0**: Standard Input (`stdin`) - Tastiera
- **1**: Standard Output (`stdout`) - Terminale
- **2**: Standard Error (`stderr`) - Terminale (per messaggi critici)

<Callout type="pro" title="Tutto è un File">
  Questa è la filosofia zen di Unix: file su disco, tastiera, monitor e persino
  la scheda di rete vengono trattati tutti come **File Descriptors**. Puoi usare
  `write()` per salvare un documento o per inviare un messaggio su internet: per
  il programmatore C, la logica è la stessa.
</Callout>

### L'Architettura: Ring 3 vs Ring 0

La CPU non si fida del tuo codice. Il tuo programma gira in **User Mode (Ring 3)**, dove non può toccare direttamente il disco o la memoria degli altri processi. Per farlo, deve invocare una **System Call**.

## Esercitazione: Programmazione a Basso Livello

<CodeEditor 
  initialCode={`#include <unistd.h> // Header per le Syscall POSIX
#include <string.h>
#include <errno.h>
#include <stdio.h>

int main() {
const char \*msg = "Scrittura diretta nel Terminale (fd 1)\\n";

    // write(file_descriptor, buffer, lunghezza)
    // Non c'è buffering: i dati partono subito!
    ssize_t written = write(1, msg, strlen(msg));

    if (written == -1) {
        // Se write fallisce, errno viene impostato dal Kernel
        perror("Errore nella write");
        return 1;
    }

    const char *err_msg = "Questo è un errore (fd 2)\\n";
    write(2, err_msg, strlen(err_msg));

    return 0;

}`}

### Riassunto Finale

Abbiamo rotto il muro della libreria standard. Ora sai che ogni volta che scrivi qualcosa a video o su disco, stai chiedendo il permesso al Kernel. Padroneggiare le syscall significa avere il controllo totale sulla macchina e capire come gira realmente il mondo Linux/Unix.
