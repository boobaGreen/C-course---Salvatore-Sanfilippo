## Oltre i limiti del tipo

Nelle lezioni precedenti abbiamo visto che i puntatori non solo sanno _dove_ si trova un dato, ma anche _di che tipo_ è. Questo è cruciale perché dice al compilatore **quanti byte leggere**.

Ma cosa succede se "inganniamo" il compilatore forzando un puntatore a leggere la memoria in un altro modo? In C, possiamo farlo usando il **Casting**.

<MemoryCaster />

### Cos'è il Casting?

Il casting è un'operazione che dice esplicitamente al compilatore di trattare una variabile come se fosse di un tipo diverso.

```c
short s = 514;          // Occupa 2 byte
char *p = (char *)&s;   // Forzo un puntatore char* a guardare dove inizia lo short
```

Se `s` vale `514` (che in esadecimale è `0x0202`), la sua memoria fisica conterrà due byte separati: `0x02` e `0x02`.
Quando leggiamo `*p`, stiamo leggendo solo **il primo byte** (`0x02`, che in decimale fa `2`), perché `p` è un puntatore a `char` (1 byte).

<Callout type="pro">
**Type Punning**
Questa tecnica di leggere la memoria di un tipo attraverso un puntatore di un altro tipo si chiama "Type Punning". È potente per la programmazione di sistema, ma può causare comportamenti indefiniti se non si rispettano le regole di allineamento.

<Callout type="danger" title="Strict Aliasing Rule">
  Il compilatore assume che due puntatori di tipo diverso (es. `int*` e `float*`) non puntino mai alla stessa zona di memoria. Se rompi questa regola con il casting, l'ottimizzatore potrebbe cancellare pezzi del tuo codice! 
  
  **Esempio**: Se cambi un `int` tramite un `char*`, il compilatore potrebbe pensare che l'`int` non sia mai cambiato e usare il vecchio valore salvato in un **Registro della CPU** invece di rileggere dalla RAM.
</Callout>
</Callout>

### Endianness: La guerra dei Byte

L'**Endianness** definisce l'ordine in cui i byte che compongono un numero multi-byte (come uno `short` a 2 byte o un `int` a 4 byte) vengono immagazzinati nella memoria fisica.

- **Little Endian** (architetture Intel/AMD x86): Il byte _meno significativo_ (Least Significant Byte - LSB) viene salvato all'indirizzo di memoria più basso.
- **Big Endian** (Protocolli di rete TCP/IP, vecchi processori Motorola): Il byte _più significativo_ (Most Significant Byte - MSB) viene salvato all'indirizzo più basso.

Esempio col numero `0x11223344`:

- In memoria **Little Endian**: `44 33 22 11` (all'indirizzo base troviamo `44`).
- In memoria **Big Endian**: `11 22 33 44` (all'indirizzo base troviamo `11`).

<Callout type="info" title="Perché Little Endian?">
  Anche se il Big Endian sembra più intuitivo per noi umani (si legge da
  sinistra a destra come un numero decimale), il **Little Endian** è spesso
  preferito dai processori perché permette di trattare un numero a 32-bit come
  uno a 8-bit semplicemente leggendo il primo byte, senza dover calcolare
  l'offset della fine del numero. È un "troncamento gratuito" a livello
  hardware!
</Callout>

<Callout type="pro" title="Network Byte Order">
  Internet parla **Big Endian**. Poiché ogni computer ha la suaEndianness,
  quando invii un numero sulla rete devi usare funzioni come `htons()` (Host TO
  Network Short) per assicurarti che il destinatario non riceva i byte al
  contrario.
</Callout>

### Lo Swap dei Puntatori: Efficienza Pura

Salvatore ci mostra un trucco fondamentale per le simulazioni (come il Game of Life): invece di copiare migliaia di byte dalla "nuova griglia" alla "vecchia griglia" a ogni turno, scambiamo semplicemente i due **indirizzi di memoria**.

```c
char *temp = old_ptr;
old_ptr = new_ptr;
new_ptr = temp;
```

È un'operazione istantanea che non dipende dalla dimensione dei dati!

## Stringhe C vs Stringhe Prefisse

Fino ad ora abbiamo usato le classiche stringhe C (note come _Null-terminated strings_). Finiscono col byte `0x00`.

Tuttavia, queste stringhe hanno un problema:

1. Per sapere la lunghezza devi scorrere tutta la stringa (complessità `O(N)`).
2. Non possono contenere `0x00` al loro interno, perché verrebbe interpretato come la fine!

Un'alternativa storica (resa celebre dal linguaggio Pascal) sono le **Length-Prefixed Strings**.
In questo formato, il **primo byte** conteneva la lunghezza della stringa.

```c
// Stringa Pascal. Il primo byte (11) dictata la lunghezza.
char *pascal_str = "\x0BHello World"; // \x0B è 11 in esadecimale.
```

In questo modo ottenere la lunghezza era un'operazione istantanea (`O(1)`), leggendo semplicemente `pascal_str[0]`. Lo svantaggio con un solo byte? Lunghezza massima di 255 caratteri!

int main() {
// Stringa stile Pascal: primo carattere = 5 (lunghezza), poi "Hello"
char str[] = {5, 'H', 'e', 'l', 'l', 'o'};

    // Leggiamo la lunghezza:
    int len = str[0];

    printf("La stringa è lunga %d caratteri.\\n", len);

    // Stampiamo i restanti caratteri
    for(int i = 1; i <= len; i++) {
        putchar(str[i]);
    }
    printf("\\n");

    // ESPERIMENTO: Casting a short
    short *s = (short *)str;
    printf("Valore del primo short in memoria esadecimale: 0x%04X\\n", s[0]);
    // Hint: In un sistema Little Endian (x86), compilerà 'H' (0x48) come byte più significativo
    // e 5 (0x05) come byte meno significativo -> 0x4805.

    return 0;

}`}
