## La Fisica della Chiamata: Calling Conventions

Quando chiami una funzione variadica, la CPU segue una **Calling Convention** (come System V su Linux o cdecl su Windows).

<Callout type="pro" title="Perché va_start richiede l'ultimo parametro fisso?">
  La CPU ha bisogno di un punto di riferimento (un'ancora). `va_start` prende
  l'indirizzo dell'ultimo parametro fisso nello stack e aggiunge un **offset**
  per trovare il primo parametro variabile. Senza quell'ancora, la funzione
  'brancolerebbe nel buio' della RAM.
</Callout>

### Registri vs Stack

Sui sistemi moderni a 64 bit (x86_64), i primi 6 argomenti passano nei **registri** (RDI, RSI, RDX...). Se la funzione è variadica, il compilatore deve spesso 'scaricare' questi registri in una zona speciale dello stack chiamata **Register Save Area** per permettere a `va_list` di navigarli.

---

## Delegazione: Il Pattern `vprintf`

Non puoi passare i tre puntini `...` da una funzione a un'altra. In C, i variadici "muoiono" nella funzione che li riceve, a meno di non usare la delegazione tramite `va_list`.

### Esempio: Logger Professionale

```c
void my_log(const char *level, const char *fmt, ...) {
    va_list ap;
    va_start(ap, fmt);
    printf("[%s] ", level);
    vprintf(fmt, ap); // Unica via per passare i variadici!
    va_end(ap);
}
```

## Sicurezza: L'Attacco Format String

<Callout type="danger" title="Regola d'oro di sicurezza">
  MAI fare `printf(user_input)`. Se un utente inserisce `%x %x %x`, vedrà il
  contenuto segreto del tuo stack (password, chiavi). Se inserisce `%n`,
  potrebbe persino sovrascrivere l'indirizzo di ritorno e prendere il controllo
  del PC. Usa sempre `printf("%s", user_input)`.
</Callout>
