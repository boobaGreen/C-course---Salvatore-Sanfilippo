# Lesson 21: Unions and Bitfields

In this lesson, Salvatore addresses the "technical debt" accumulated while talking about structures: **Unions** and **Bitfields**. These tools provide even more granular control over memory, allowing us to overlap variables or use individual bits.

## What is a Union?

While in a `struct` every field has its own unique address, in a `union` all fields start at the same **offset 0**.

<ComparisonTable 
  title="Struct vs Union"
  headers={["Feature", "Struct", "Union"]}
  rows={[
    ["Memory", "Sum of field sizes (plus padding)", "Size of the largest field"],
    ["Access", "All fields are valid simultaneously", "Only one field is 'active' at a time"],
    ["Offset", "Each field has its progressive offset", "All fields start at the same address"]
  ]}
/>

## Bitfields: Saving the Impossible

Imagine you need a value that only goes from 0 to 15. You only need 4 bits, but a `char` takes up 8. With Bitfields, you can tell C to use only the necessary space.

<Infographic 
  title="Anatomy of a Bitfield"
  description="Packing multiple values into a single byte."
  items={[
    { label: "Field A", value: "4 bits (0-15)", color: "blue" },
    { label: "Field B", value: "4 bits (0-15)", color: "emerald" },
    { label: "Total", value: "1 Byte!", color: "amber" }
  ]}
/>

<Callout type="warning" title="Portability">
How bits are packed (order and alignment) is not guaranteed by the ANSI C standard and depends on the compiler and architecture. Salvatore recommends using them with caution for rigid network protocols.
</Callout>

### Final Summary
We have learned to squeeze every single bit from our RAM. `unions` allow us to see memory from different angles, while `bitfields` give us surgical control over space. Together, these techniques make C the language of choice for high-performance systems like Redis.
