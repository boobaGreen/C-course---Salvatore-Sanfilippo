## The Physicality of the Union

While in a `struct` every field has its own unique address, in a `union` all fields start at the same **offset 0**. It's like having a room with multiple labels on the door: there is only one occupant, but you can call them by different names.

## Bitfields: Programming the Silicon

Bitfields allow you to map individual bits of a byte to variable names. This is the daily bread of those who write **Drivers** or **Network Protocols**.

<Callout type="pro" title="Example: Hardware Register">
  Imagine a chip that controls LEDs. Its register is a byte: bits 0-3 control the color, bit 4 is the power switch.
  ```c
  struct LED_Register {
      unsigned char color  : 4; // only 4 bits
      unsigned char active : 1; // 1 bit (0 or 1)
      unsigned char unused : 3; // 3 padding bits
  };
  ```
</Callout>

### The Portability Risk

<Callout type="danger" title="Beware of Bit-Endianness">
  Not only can bytes be flipped (Little/Big Endian), but the bit packing order inside a bitfield also depends on the compiler! If you exchange a struct with a bitfield between an x86 PC and an ARM controller, the bits might be read "backwards." For network code, always use bitwise operators (`&`, `|`, `<<`) which are universal.
</Callout>

### Final Summary

We have learned to squeeze every single bit from our RAM. `unions` allow us to see memory from different angles, while `bitfields` give us surgical control over space. Together, these techniques make C the language of choice for high-performance systems like Redis and for direct hardware control.
