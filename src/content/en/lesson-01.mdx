## The "Hello World" in C

C is a very old language that has evolved relatively little in its core syntax, but immensely in its **compilers**. It remains a language made of "few ideas", which must however be assimilated thoroughly. Being a **compiled language**, we cannot execute the source code directly: we must first translate it into machine language.

Here is the classic starting program, in just 5 lines:

```c
#include <stdio.h>

int main() {
    printf("Hello, World!\n");
    return 0;
}
```

<TypeMatcher />

### How is it compiled and executed?

C is a **compiled** language. This means that the code we write is not read line by line by an interpreter (like Python or JS), but transformed entirely into a binary file before it can be launched.

If we save this code in a file called `hello.c`, we cannot execute it as it is. We must use the **C compiler** (like `cc`, `gcc`, or `clang`):

<TerminalSimulation command="cc hello.c" output="" />

This command generates a new executable file (typically called `a.out` on Unix systems). This file is a **binary** containing the machine language and some specific headers of the operating system. We can execute it by typing:

<TerminalSimulation command="./a.out" output="Hello, World!" />

## Behind the scenes: Assembly and Optimizations

We can ask the compiler to stop before generating the final binary, allowing us to see the intermediate **Assembly** code, using the `-S` flag:

<TerminalSimulation 
  command="cc -S hello.c" 
  language="asm"
>
{`	.file	"hello.c"
	.text
	.section	.rodata
.LC0:
	.string	"Hello, World!"
	.text
	.globl	main
	.type	main, @function
main:
.LFB0:
	.cfi_startproc
	pushq	%rbp
	.cfi_def_cfa_offset 16
	.cfi_offset 6, -16
	movq	%rsp, %rbp
	.cfi_def_cfa_register 6
	leaq	.LC0(%rip), %rdi
	call	puts@PLT
	movl	$0, %eax
	popq	%rbp
	.cfi_def_cfa 7, 8
	ret
	.cfi_endproc`}
</TerminalSimulation>

Looking at the Assembly, we can notice that the string "Hello, World!" is defined and the `printf` function is called. The beauty of modern C, however, lies in the **optimization levels**.

If we compile with the `-O2` flag (level 2 optimization):
<TerminalSimulation 
  command="cc -O2 -S hello.c" 
  language="asm"
>
{`	.file	"hello.c"
	.text
	.section	.rodata.str1.1,"aMS",@progbits,1
.LC0:
	.string	"Hello, World!"
	.section	.text.startup,"ax",@progbits
	.p2align 4
	.globl	main
	.type	main, @function
main:
.LFB23:
	.cfi_startproc
	subq	$8, %rsp
	.cfi_def_cfa_offset 16
	leaq	.LC0(%rip), %rdi
	call	puts@PLT
	xorl	%eax, %eax
	addq	$8, %rsp
	.cfi_def_cfa_offset 8
	ret
	.cfi_endproc`}
</TerminalSimulation>

We will notice something surprising: **the `printf` function disappears**! The compiler, realizing that we are printing a simple string followed by a newline (`\n`), automatically substitutes `printf` with the `puts` function, which is faster and more efficient for this specific purpose, removing the original `\n` because `puts` adds it by itself.
This is just a "drop in the ocean" of the optimizations that a mature compiler can do.

## Tearing down the program

Let's analyze each line to understand what really happens:

### 1. The Preprocessor and `#include`
```c
#include <stdio.h>
```
This line is not strictly part of the C language itself, but is a **preprocessor directive** (all directives start with `#`). Before the actual compilation, the preprocessor takes the content of the `stdio.h` file (a standard library file) and "unpacks" it directly inside our file.

Why do we need it? The C language itself has very few keywords (`return`, `int`, `while`). Functions like `printf` are not part of the language, but of the **Standard Library**. If we didn't include `stdio.h` (Standard Input/Output Header), the compiler wouldn't know what `printf` is!

### 2. Header Files (`.h`) and Function Prototypes

In header files (like `stdio.h`) there is no code that *executes* the `printf`, but there are **function prototypes**.

<ComparisonTable 
  title="Prototype vs Definition"
  headers={["Feature", "Prototype (.h)", "Definition (.c)"]}
  rows={[
    ["Purpose", "Declares the existence and signature of the function", "Contains the actual code implementation"],
    ["Syntax", "Ends with a semicolon: <br/><code>int sum(int a, int b);</code>", "Contains the body in braces: <br/><code>int sum(int a, int b) { ... }</code>"],
    ["Visibility", "Allows other files to know how to call the function", "Gets compiled into the final binary object"]
  ]}
/>

A prototype is similar to a function definition, but instead of having a block of code between curly braces `{}`, it ends immediately with a semicolon `;`:

```c
int printf(const char *format, ...);
```

The prototype tells the compiler: *"There is a function called `printf` that accepts these parameters and returns an integer. Trust me, you'll find it later"*.

Thanks to the prototype, the compiler knows exactly how to prepare data in memory (Assembly) to call it correctly, even without knowing its internal code. In fact, we could even skip `#include <stdio.h>` and write the prototype by hand in our file to make the program work!

<KeyConcepts title="To recap">
  - C is not interpreted, but **compiled**.
  - The compiler can optimize code intelligently (e.g., transforming `printf` into `puts`).
  - The **preprocessor** prepares the file before compilation.
  - **Header (`.h`)** files contain function **prototypes**, essential to tell the compiler how to call functions written elsewhere.
</KeyConcepts>
