## "Hello World" in C

C is a very old language that has evolved relatively little in its basic syntax, but tremendously in its **compilers**. It remains a language of "few ideas" that must be deeply assimilated. Being a **compiled language**, we cannot execute the source code directly: we must first translate it into machine language.

Here is the classic starting program, in just 5 lines:

```c
#include <stdio.h>

int main() {
    printf("Hello, World!\n");
    return 0;
}
```

<Callout type="tip" title="Salvatore's Tip">
  Don't try to understand everything at once. In C, many concepts become clear only when we go a level deeper and see what happens in RAM or the CPU.
</Callout>

### How to compile and execute?

C is a **compiled** language. This means the code we write isn't read line-by-line by an interpreter (like Python or JS), but fully transformed into a binary file before it can be run.

![The Compilation Pipeline](/images/c_compilation_pipeline_infographic.png)

If we save this code in a file called `hello.c`, we cannot execute it as it is. We need to use the **C compiler** (like `cc`, `gcc`, or `clang`):

<TerminalSimulation command="cc hello.c" output="" />

This command generates a new executable file (typically named `a.out` on Unix systems). This file is a **binary** containing machine language and some operating system-specific headers (Linux, macOS, Windows). We can run it by typing:

<TerminalSimulation command="./a.out" output="Hello, World!" />

## Behind the Scenes: Assembly and Optimizations

We can ask the compiler to stop before generating the final binary, showing us the intermediate **Assembly** code using the `-S` flag.

Assembly is the "near-machine" language: it's readable by humans but corresponds 1:1 to the instructions the CPU will execute.

<AssemblyExplorer />

<Callout type="hacker" title="Hacker Mindset">
  Looking at Assembly isn't an academic exercise. It's the only way to *truly* understand how efficient your code is and what the compiler is doing behind your back.
</Callout>

As you saw in the explorer above, if we compile with the `-O2` flag (optimization level 2), we'll notice something surprising: **the `printf` function disappears**!

The compiler, realizing we are printing a simple string followed by a newline (`\n`), automatically replaces `printf` with the `puts` function, which is faster and more efficient for this specific purpose. This is just a "drop in the ocean" of the optimizations that a mature compiler can do.

## Breaking Down the Program

Let's analyze each line to understand what's really happening:

### 1. The Preprocessor and `#include`
```c
#include <stdio.h>
```
This line is not part of the C language itself, but a **preprocessor directive** (all directives start with `#`). Before actual compilation, the preprocessor takes the content of the `stdio.h` file (a standard library file) and "unpacks" it directly into our file.

Why do we need it? The C language itself has very few keywords (`return`, `int`, `while`). Functions like `printf` are not part of the language, but of the **Standard Library**. If we didn't include `stdio.h` (Standard Input/Output Header), the compiler wouldn't know what `printf` is!

<Callout type="info">
  C is a "naked" language. Without libraries, it doesn't even know how to print a character to the screen. Everything "useful" is found in the Standard Library.
</Callout>

### 2. Header Files (`.h`) and Function Prototypes

In a header file (like `stdio.h`), there isn't the code that *executes* `printf`, but there are **function prototypes**.

<ComparisonTable 
  title="Prototype vs Definition"
  headers={["Feature", "Prototype (.h)", "Definition (.c)"]}
  rows={[
    ["Purpose", "Declares the existence and signature of the function", "Contains the actual code implementation"],
    ["Syntax", "Ends with a semicolon: <br/><code>int sum(int a, int b);</code>", "Contains the body between braces: <br/><code>int sum(int a, int b) { ... }</code>"],
    ["Visibility", "Allows other files to know how to call the function", "Compiled into the final binary object"]
  ]}
/>

A prototype is similar to a function definition, but instead of having a block of code between curly braces `{}`, it ends immediately with a semicolon `;`:

```c
int printf(const char *format, ...);
```

A prototype tells the compiler: *"There exists a function called `printf` that accepts these parameters and returns an integer. Trust me, you'll find it later"*.

Thanks to the prototype, the compiler knows exactly how to prepare data in memory (Assembly) to call it correctly, even without knowing its internal code. In fact, we could even skip `#include <stdio.h>` and write the prototype by hand in our file to make the program work!

<KeyConcepts title="Recap">
  - C is not executed, but **compiled**.
  - The compiler can optimize code intelligently (e.g., turning `printf` into `puts`).
  - The **preprocessor** prepares the file before compilation.
  - **Header files (`.h`)** contain function **prototypes**.
</KeyConcepts>
