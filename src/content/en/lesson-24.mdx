## The Stack in C

In ToyFort, every C function receives a `TFContext`. This structure contains the data "Stack." When you write `+`, the corresponding C function must:

1. Verify there are at least 2 elements.
2. Extract (POP) the values.
3. Perform the operation.
4. Insert (PUSH) the result.

<Callout type="pro" title="Ownership Management">
  Every `pop` from the stack gives you ownership of the object. If you forget to
  call `tf_object_release()`, you've created a **Memory Leak**. If you call it
  twice, you have a **Double Free**. Discipline is the C programmer's only
  defense.
</Callout>

## Amortized O(1): The Secret of Fast Lists

Why doesn't Salvatore allocate exactly the space needed for every new element?
If we had to call `realloc()` for every single element, we'd waste time copying data every time. Instead, we double the capacity whenever it gets full.

## Implementation: The Heart of Eval

<CodeEditor 
  initialCode={`#include <stdio.h>
#include <string.h>

void tf_eval(TFContext *ctx, TFObject *obj) {
if (obj->type == TF_INT || obj->type == TF_STR || obj->type == TF_LIST) {
// Literal value: push it onto the stack
tf_stack_push(ctx, tf_object_retain(obj));
} else if (obj->type == TF_SYM) {
// Symbol: look it up in the dictionary
TFFunc f = tf_dict_lookup(ctx, obj->s);
if (f) {
f(ctx); // Execute the associated C function
} else {
fprintf(stderr, "Error: Unknown symbol '%s'\\n", obj->s);
}
}
}
`}
language="c"
/>

<Callout type="warning" title="Branch Prediction">
  The `eval` loop is a challenge for the CPU. Because the called functions
  change constantly, the **Branch Predictor** struggles to predict where the
  code will jump. This "dispatch overhead" is the price of flexibility in an
  interpreter compared to compiled code.
</Callout>
