## The Physics of the Call: Calling Conventions

When you call a variadic function, the CPU follows a **Calling Convention** (like System V on Linux or cdecl on Windows).

<Callout type="pro" title="Why does va_start require the last fixed parameter?">
  The CPU needs a reference point (an anchor). `va_start` takes the address of
  the last fixed parameter on the stack and adds an **offset** to find the first
  variable parameter. Without that anchor, the function would be 'groping in the
  dark' within RAM.
</Callout>

### Registers vs. Stack

On modern 64-bit systems (x86_64), the first 6 arguments are typically passed in **registers** (RDI, RSI, RDX...). If the function is variadic, the compiler must often 'dump' these registers into a special stack area called the **Register Save Area** to allow `va_list` to navigate them.

## Delegation: The `vprintf` Pattern

You cannot pass the three dots `...` from one function to another. In C, variadics "die" in the function that receives them, unless you use delegation via `va_list`.

### Example: Professional Logger

```c
void my_log(const char *level, const char *fmt, ...) {
    va_list ap;
    va_start(ap, fmt);
    printf("[%s] ", level);
    vprintf(fmt, ap); // The only way to pass variadics!
    va_end(ap);
}
```

## Security: The Format String Attack

<Callout type="danger" title="Golden Rule of Security">
  NEVER do `printf(user_input)`. If a user enters `%x %x %x`, they will see the
  secret content of your stack (passwords, keys). If they enter `%n`, they could
  even overwrite the return address and take control of the PC. Always use
  `printf("%s", user_input)`.
</Callout>

### Final Summary

We've explored the bridge between C's extreme flexibility and the physical rigidity of the CPU stack. Variadic functions are a powerful but "blind" tool: it's up to you, as a conscious programmer, to guide them carefully to avoid memory leaks and catastrophic security holes.
