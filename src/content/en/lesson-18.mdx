## Using Typedef

`typedef` does not create a new type but a "nickname" for an existing type. Salvatore highly recommends it to avoid writing `struct` every time.

```c
typedef struct {
    int x;
    int y;
} Point;

Point p; // Much cleaner than 'struct Point p'
```

<Callout type="info" title="Why not use typedef everywhere?">
Salvatore notes that while convenient, overusing `typedef` for base types (e.g., `typedef int error_code`) can hide the nature of the data and make the code harder to read for those unfamiliar with your aliases. Use it wisely!
</Callout>

### Anonymous Structs
Notice that in the `typedef` above, we didn't give the struct a name after the `struct` keyword. This is an **Anonymous Struct**: we only need the alias name `Point` to use it in our program.

## The File API (stdio.h)

In C, a file is seen as a **stream** of bytes. The standard library provides us with a pointer to a special structure called `FILE`.

<Callout type="hacker" title="Opaque Types">
The `FILE` type is the first example of an **Opaque Type** we encounter. We don't know (and shouldn't know) what's inside the `FILE` structure. We only use the pointer and the functions (`fopen`, `fread`, etc.) to manipulate it. This is design excellence: hiding internal complexity.
</Callout>

### Example: Writing and Reading a Binary

<CodeEditor 
  initialCode={`#include <stdio.h>
#include <stdint.h>

typedef struct {
    uint32_t id;
    float score;
} Player;

int main() {
    Player p1 = {1, 95.5f};
    
    // Writing
    FILE *fout = fopen("save.dat", "wb");
    if (fout) {
        fwrite(&p1, sizeof(Player), 1, fout);
        fclose(fout);
    }
    
    // Reading
    Player loaded;
    FILE *fin = fopen("save.dat", "rb");
    if (fin) {
        fread(&loaded, sizeof(Player), 1, fin);
        printf("ID: %u, Score: %.1f\\n", loaded.id, loaded.score);
        fclose(fin);
    }
    
    return 0;
}`}

### Final Summary
We gave a name to our structures and persistence to our data. `typedef` and the `FILE` API are the foundations upon which you will build software that doesn't die when the terminal is closed, but can exchange data with the entire digital ecosystem.
