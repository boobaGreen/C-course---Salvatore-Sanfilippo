## Loops: for vs while

As Salvatore explains, a `for` loop is often just a more compact way of writing a `while`.

```c
// WHILE Version
int i = 0;
while (i < 5) {
    // code
    i++;
}

// FOR Version (cleaner)
for (int i = 0; i < 5; i++) {
    // code
}
```

<Callout type="info" title="Syntactic Sugar">
  The `for` loop is what programmers call "Syntactic Sugar": it doesn't add new
  functionality to the language, it just makes a structure you could already
  write with `while` "sweeter" (more readable and compact).
</Callout>

### Transformation: From For to While

To truly understand the `for`, learn to dismantle it:

1. The first part (initialization) goes **before** the loop.
2. The second part (condition) is the **engine** of the `while`.
3. The third part (increment) goes at the **end** of the code block.

We use `for` when we already know how many times to iterate, and `while` when the condition depends on something that will happen in the code.

<Callout type="pro" title="For: Total Flexibility">
In C, the three parts of the `for` (initialization, condition, increment) are optional!
- `for(;;)` is an idiomatic way to write an **infinite loop**.
- You can omit the increment if you handle it inside the loop body.

<Callout type="warning" title="Loop Unrolling">
  The compiler can decide to "unroll" a short `for` loop (e.g., 4 iterations) by writing the 4 instructions directly one after the other. This eliminates the cost of checking the condition `i < 4` at each turn and helps the CPU's **Pipeline** flow without branch-induced interruptions.
</Callout>
</Callout>

### Practical Example: Array Search

Salvatore shows how to use `break` to stop a loop as soon as we find what we're looking for.

```c
int a[] = {1, 5, 8, 4, 0}; // 0 is the terminator
int i;
for (i = 0; a[i] != 0; i++) {
    if (a[i] == 8) break; // Found it! Exit immediately.
}

if (a[i] == 0) printf("Not found\n");
else printf("Found at position %d\n", i);
```

<Callout type="warning" title="Variable Scope of i">
  If you declare `for (int i = 0; ...)` inside the loop, the variable `i`
  disappears as soon as the loop ends. If you need to know where the loop
  stopped (like in the search example above), you must declare `i` **outside**
  the `for`.
</Callout>

<Callout type="danger" title="The 'Dangling Else' Problem">
  When you nest `if` statements without using braces, the `else` always binds to
  the nearest `if`. This is one of C's trickiest bug sources. **Salvatore's
  Advice:** always use braces `{}`, even for a single line!
</Callout>

## Recursion and the Stack

When a function calls itself, the computer doesn't start from scratch. It "freezes" the current function and opens a new one on top of it (a new "frame").

<Callout type="pro" title="Tail Call Optimization (TCO)">
If the last thing a function does is call itself, a good compiler can optimize it by turning it into a jump (`goto`), saving memory.

**Important**: TCO only works if there is no "residual work".

- `return f(x);` -> Optimizable.
- `return 1 + f(x);` -> **NOT** optimizable (the CPU must remember to add 1 after the function returns).

Salvatore showed us how to do it manually to avoid consuming the stack:

```c
loop:
    if (start > end) return;
    printf("%d ", start);
    start = start + 1;
    goto loop; // Manual recursion (iteration)!
```

In this way, the memory address of `start` will always remain the same, because we don't create new frames.

</Callout>

<RecursionVisualizer />

<Callout type="warning" title="Stack Overflow">
  If recursion has no "exit condition" (base case), it will keep creating new
  frames until it runs out of memory dedicated to the stack. This causes the
  famous 'Stack Overflow' crash.
</Callout>

## Choosing Elegantly: The Switch

If you have a variable that can be 1, 2, 3... instead of writing ten `if`s, we use `switch`. It's more readable and often faster!

<SwitchBoard />

<Callout type="pro" title="Jump Tables">
Under the hood, if the cases are close together (e.g., 1, 2, 3), the compiler doesn't do many `if` checks. It creates a **Jump Table**: an array of memory addresses. The processor takes the value, uses it as an index in the table, and jumps directly to the correct code. It's incredibly fast!

**The Limit**: If cases are very far apart (e.g., `case 1` and `case 1000000`), the compiler cannot create such a large table. In that case, it "falls back" to using a series of `if`s or a binary search.

</Callout>

<Callout type="info" title="The Importance of Break">
  In a `switch`, if you forget the `break;`, the code will continue to execute
  subsequent cases as well! This behavior is called 'fall-through'.
</Callout>

## Real Code: Redis Glob Matching

Salvatore shows how he uses these concepts in real production code. His string matching function (e.g., handles `*`, `?`, `[a-z]`) uses both `switch` and recursion to handle various pattern characters.

```c
// Simplified version of Salvatore's glob matching
int glob_match(const char *pattern, const char *string) {
    switch (pattern[0]) {
        case '*': // Recursion to handle the asterisk
            // ... complex logic ...
            break;
        case '?': // Skip one character
            // ...
            break;
        default: // Literal match
            if (pattern[0] != string[0]) return 0;
    }
}
```
