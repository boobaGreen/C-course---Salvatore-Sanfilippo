## Object Architecture: TFObject

To represent a language, we need a universal data type. In ToyFort, we use a `struct` that encloses a `union`.

<Callout type="pro" title="Hardware Efficiency">
  By using a `union` inside `TFObject`, we ensure that each object only takes up
  the memory of the largest type (usually the list or string), regardless of
  whether it contains just a small integer. This optimization allows languages
  like Python or Ruby to manage millions of objects.
</Callout>

### Anatomy of a Heterogeneous List

The `TFObject **ele` field is fundamental: it's an array of pointers. Why not an array of objects?

1. **Heterogeneity**: Each cell can point to a different object type.
2. **Sharing**: Multiple lists can point to the same object (thanks to `ref_count`).
3. **Recursion**: A list can contain another list, creating execution trees.

## Stack-based Logic: RPN (Reverse Polish Notation)

Forget parentheses. In ToyFort, you write `5 5 +`.

1. The CPU (via our interpreter) pushes 5 onto the stack.
2. It pushes another 5.
3. When it encounters `+`, it pops the last two, adds them, and pushes the result back.

fprintf(stderr, "FATAL: Out of memory\\n");

return p;
}

typedef enum { TF_INT, TF_STR, TF_LIST } TFType;

struct TFObject {
TFType type;
int ref_count;
union {
long long i;
char \*s;
struct {
struct TFObject \*\*ele;
int len;
} list;
};
};
