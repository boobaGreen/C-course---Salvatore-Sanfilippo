## Why Structs? The Fraction Example

Imagine you want to manage thousands of fractions. Without structs, you would need thousands of `num` and `den` variables. A nightmare. Salvatore shows us how a `struct` solves the problem:

```c
struct Fraction {
    int num;  // Numerator
    int den;  // Denominator
};
```

### Anatomy of an Object in Memory

A struct is a contiguous block of memory. If an `int` occupies 4 bytes, our `struct Fraction` will occupy exactly **8 bytes**.

## The Mystery of Padding (Advanced Knowledge)

What happens if we add a `char` (1 byte) to our struct? We would expect a size of 9 bytes (4 + 4 + 1). But C surprises us: the size will be **12 bytes**.

<Callout type="pro" title="Why does the CPU hate misalignment?">
  A 64-bit processor reads RAM in complete 8-byte blocks. If an `int` is saved
  halfway between one block and another (misaligned), the CPU must perform **two
  reads** instead of one, bit-shift the data, and merge the results. Padding
  sacrifices a little RAM to avoid this massive hardware slowdown.
</Callout>

<Callout type="warning" title="Optimization">
To save memory, Salvatore suggests grouping fields by size: first the largest (pointers, long, int) and then the smallest (char). This reduces the padding inserted by the compiler!

<Callout type="pro" title="Attribute Packed">
  In extreme cases (e.g., net packets that must be identical to the millimeter), you can tell GCC: `__attribute__((packed))`. This forces the compiler to remove **all** padding. Warning: the program will become slightly slower because the CPU will have to work harder to read misaligned data.
</Callout>
</Callout>

<Callout type="info" title="The Origin of the Arrow ->">
  Why not just use the dot? In the early years of C, the compiler needed a
  strong syntactic hint to know if you were accessing a fixed location (dot) or
  if it had to first calculate a base address from a register (arrow). Today,
  it's a stylistic choice that makes code extremely readable: you see an arrow,
  you know a pointer is involved.
</Callout>

## Operations: Dot vs Arrow

It's the classic confusion for every C student. Here is Salvatore's definitive rule dug out of the language's philosophy:

- **Dot Operator (`.`)**: Used when you have the **physical variable** (on the stack or as a copy).
- **Arrow Operator (`->`)**: Used when you have a **pointer** (memory address). It is shorthand for `(*ptr).field`.

<Callout type="pro" title="Mapping Hardware: Bitfields">
  C allows you to map individual bits of a struct to speak directly to hardware (e.g., video cards or sensors):
  ```c
  struct Status {
      unsigned int power_on : 1; // Occupies only 1 BIT
      unsigned int error    : 1; // Occupies only 1 BIT
      unsigned int code     : 6; // Occupies 6 BITS
  }; // Total: only 1 byte instead of 4-12!
  ```
</Callout>

### Final Summary

Structures are the beating heart of professional C. They allow us to model reality (fractions, users, network packets) respecting the hard logic of physical memory. Remember: field order matters for padding, and the arrow is the key to manipulating dynamic objects!
