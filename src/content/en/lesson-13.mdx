## The Problem with Pascal Strings

In the last lesson, we saw prefixed length strings. They worked, but they had a fatal flaw: if you passed a Pascal string to `printf`, it would "go crazy" because the first byte (the length) was interpreted as a character.

Salvatore teaches us the **veteran's trick**: move the prefix **before** the memory address we give to the user.

<Callout type="warning" title="Metadata Security">
  Saving the length *before* the pointer protects our system. If a malicious
  user tries to send an overly long string via `strcpy`, they will overwrite
  data *after* the string, but they won't touch the length (which is *before*),
  making it harder to corrupt the program's logical structure. It's a primitive
  form of a **Security Canary**.
</Callout>

### Why `uint32_t` from `<stdint.h>`?

Salvatore is categorical: we cannot use a simple `int`.
An `int` on a 16-bit system occupies 2 bytes, on a 32-bit one it's 4. If we want our strings to be portable and our header to **always** occupy the same physical space in RAM and on disk, we must use fixed-size types. This is the standard for any **Binary Protocol** (like network packets or game save files).

### Towards Structs

Manually manipulating offsets (like `s - 4` or `s - 8`) is "fragile" and error-prone. This is why Salvatore introduces **Structs**: a way to group different data (length, refcount, text) into a single logical entity.

```c
struct ps_string {
    uint32_t len;
    uint32_t ref_count;
    char buf[]; // Flexible Array Member (C99)
};
```

---

## Performance Simulation: O(N) vs O(1)

Imagine a string 10 million characters long.

- `strlen` (Standard C) must read **10,000,000 bytes**, loading them sequentially from the memory bus into CPU registers to find the trailing zero.
- `ps_len` (Ours) must read **only 1 number** at a fixed offset.

**Time difference:** If `strlen` takes 2 milliseconds, `ps_len` takes about 0.5 nanoseconds. That's a **4-million-fold** difference!

### Final Summary

Today we dug beneath the surface. We learned that a pointer is a compass: if we know where we're pointing, we can move forward to see data or backward to see metadata. This technique is at the heart of databases like Redis and many high-level languages (like Python) that need to manage strings efficiently.
