## The Problem with Pascal Strings

In the last lesson, we saw prefixed length strings. They worked, but they had a fatal flaw: if you passed a Pascal string to `printf`, it would "go crazy" because the first byte (the length) was interpreted as a character.

Salvatore teaches us the **veteran's trick**: move the prefix **before** the memory address we give to the user.

### Why `uint32_t` from `<stdint.h>`?

Salvatore is categorical: we cannot use a simple `int`.
An `int` on a 16-bit system occupies 2 bytes, on a 32-bit one it's 4. If we want our strings to be portable and our header to **always** occupy the same space, we must use fixed-size types.

## Implementing the `ps_string` Library

Here is the complete code discussed by Salvatore, including the logic for **creation**, **O(1) access**, and **correct freeing**.

<CodeEditor 
  initialCode={`#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <stdint.h>

// Define a 4-byte header (uint32_t)
#define HDR_SIZE sizeof(uint32_t)

// Create an evolved string from a C string
char *ps_create(char *init) {
    uint32_t len = strlen(init);
    // Allocate: Length (4) + Text + '\\0' (1)
    char *p = malloc(HDR_SIZE + len + 1);
    if (!p) return NULL;

    // Write the length at the start
    *((uint32_t*)p) = len;

    // Point to the text (skip the header)
    char *s = p + HDR_SIZE;
    // Copy the text (including C compatibility)
    strcpy(s, init);
    
    return s;
}

// Return length in CONSTANT O(1) time
uint32_t ps_len(char *s) {
    // Go back 4 bytes and read the value
    return *((uint32_t*)(s - HDR_SIZE));
}

// Correctly free the memory
void ps_free(char *s) {
    // CRITICAL: We must free from the ORIGINAL address (p)
    free(s - HDR_SIZE);
}

int main() {
    char *my_str = ps_create("Salvatore C Course");
    
    // Interoperability: printf sees a normal C string!
    printf("Text: %s\\n", my_str);
    
    // Power: We know the length without scanning the string
    printf("Length (O(1)): %u\\n", ps_len(my_str));
    
    ps_free(my_str);
    return 0;
}`} 
  language="c" 
/>

<Callout type="hacker" title="Reference Counting (Teaser)">
Towards the end of the lesson, Salvatore introduces an advanced concept: adding **another 4 bytes** in the header for the `ref_count`. This allows multiple variables to point to the same string without duplicating it in memory. When a variable "stops" using it, it decrements the counter. Only when the counter reaches zero is `free` actually called.
</Callout>

## Performance Simulation: O(N) vs O(1)

Imagine a string 10 million characters long.
- `strlen` (Standard C) must read **10,000,000 bytes** to find the trailing zero.
- `ps_len` (Ours) must read **only 1 number** at a fixed offset.

**Time difference:** From milliseconds to nanoseconds. On a large scale, this changes everything.

### Final Summary
Today we dug beneath the surface. We learned that a pointer is a compass: if we know where we're pointing, we can move forward to see data or backward to see metadata. This technique is at the heart of databases like Redis and many high-level languages (like Python) that need to manage strings efficiently.
