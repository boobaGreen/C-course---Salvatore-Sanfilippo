## The Execution Cycle: `execute_program`

The ToyFort interpreter loops over a list. If it finds a value, it "pushes" it onto the stack; if it finds a symbol, it executes it.

<StackVisualizer />

<CodeEditor 
  initialCode={`void execute_program(TFContext *ctx, TFObject *prg) {
    assert(prg->type == TF_LIST);
    for (size_t i = 0; i < prg->l.len; i++) {
        TFObject *word = prg->l.ele[i];
        if (word->type == TF_SYM) {
            call_symbol(ctx, word);
        } else {
            // If not a symbol, it's a value: push it onto the stack
            retain_object(word); // Increment ref_count!
            list_push(ctx->stack, word);
        }
    }
}`}

## Memory Management: Retain and Release

In C, managing memory for an interpreter is complex because objects can be shared. Salvatore implements a **Reference Counting** system.

<ComparisonTable 
  title="Manual Management vs Ref Counting"
  headers={["Method", "Advantage", "Disadvantage"]}
  rows={[
    ["Classic Malloc/Free", "Maximum speed", "High risk of double-free or leaks"],
    ["Reference Counting", "Semi-automatic management", "Memory overhead and risk of cycles"],
    ["Garbage Collector", "Fully automatic", "'Stop the world' pauses and complexity"]
  ]}
/>

### How it works in code:

```c
void retain_object(TFObject *o) {
    o->ref_count++;
}

void release_object(TFObject *o) {
    o->ref_count--;
    if (o->ref_count == 0) {
        free_object(o); // Truly deallocates memory
    }
}
```

<Callout type="warning" title="The Pop Trap">
When you extract an object from the stack with a `pop`, you become the sole owner. If you forget to call `release` at the end of your function, that object will never be freed!
</Callout>
