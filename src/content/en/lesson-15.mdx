## The Algorithmic "Shock"

Until now, we've used arrays. Arrays are flat memory blocks. But what if we don't know how many lines a file has? Instead of allocating a giant array "randomly," we use a **Linked List**.

A linked list is like a chain of people: each one holds the hand of the next one. If you want to reverse the order of lines (as the `tac` command does), you just add each new line "in front" of the head of the list.

### The Node Structure

```c
struct Line {
    char *text;          // Line content
    struct Line *next;   // Pointer to the next line (or NULL)
};
```

  Note that `struct Line` contains a pointer to `struct Line`. This is possible

  the full size of the struct to define the pointer.
</Callout>

<Callout type="info" title="Head Insertion">
Why does the list reverse the file? Because we do `new_node->next = head; head = new_node;`. Imagine stacking plates: the last one you put down is the first one you pick up. This structure is called a **Stack**.
</Callout>

<ComparisonTable
  title="Array vs Linked List"
  headers={["Feature", "Array", "Linked List"]}
  rows={[
    ["Allocation", "Fixed contiguous block", "Scattered nodes in heap"],
    ["Random Access", "O(1) - Instant", "O(N) - Must traverse"],
    ["Head Insertion", "O(N) - Must shift all", "O(1) - Change 2 pointers"],
    ["Memory Waste", "Zero (data only)", "High (each node has a pointer)"],
    ["Cache Locality", "⭐ Excellent (Close data)", "❌ Poor (Scattered data)"],
  ]}
/>

<Callout type="pro" title="The Silent Killer: Cache Misses">
  At the hardware level, the CPU doesn't read a single byte from RAM, but an
  entire **Cache Line** (usually 64 bytes). - **Array**: When you read `arr[0]`,
  the CPU also fetches `arr[1]`, `arr[2]`, etc. for free into the processor.
  Access is lightning fast. - **Linked List**: Since nodes are scattered in the
  Heap ("Pointer Chasing"), every jump to `next` might require a new slow read
  from physical RAM. On modern hardware, traversing an array can be **10-100
  times faster** than traversing a linked list!
</Callout>

## Implementation: The TAC Command

<CodeEditor 
  initialCode={`#include <stdio.h>
#include <stdlib.h>
#include <string.h>

struct Line {
char *text;
struct Line *next;
};

int main(int argc, char \*\*argv) {
if (argc < 2) return 1;

    FILE *fp = fopen(argv[1], "r");
    if (!fp) return 1;

    char buf[1024];
    struct Line *head = NULL;

    // Read the file and create the list "backwards"
    while (fgets(buf, sizeof(buf), fp)) {
        struct Line *new_node = malloc(sizeof(struct Line));
        new_node->text = strdup(buf); // Allocates and copies the string
        new_node->next = head;        // Old 'head' becomes 'next'
        head = new_node;              // This becomes the new 'head'
    }

    // Print the list (which is now in reverse order!)
    struct Line *curr = head;
    while (curr) {
        printf("%s", curr->text);
        curr = curr->next;
    }

    // CHALLENGE: Add a final loop to 'free()' each node and its string!

    fclose(fp);
    return 0;

}`}
language="c"
/>

<Callout type="info">
  `strdup` is a very handy function that does `malloc` + `strcpy` in one go.
  Remember, however, that you must `free` both the `text` and the node itself!
</Callout>

<Callout type="danger" title="Dangling Pointers">
  When you free a node with `free(curr)`, the pointer `curr` still contains the
  memory address. Using it after `free` is catastrophic (**Use-After-Free**).
  Good practice: assign `curr = NULL;` after finishing. **Fragmentation**:
  Creating thousands of small nodes with `malloc` creates many "holes" in RAM
  (external fragmentation). In embedded systems with very little memory, this
  can stall the system even if there is technically enough total space.
</Callout>
