## Beyond Data Types

In previous lessons, we've seen that pointers not only know _where_ data is located, but also _what type_ it is. This is crucial because it tells the compiler **how many bytes to read**.

But what happens if we "trick" the compiler by forcing a pointer to read memory in a different way? In C, we achieve this using **Casting**.

<MemoryCaster />

### What is Casting?

Casting is an operation that explicitly tells the compiler to treat a variable as if it were a different type.

```c
short s = 514;          // Occupies 2 bytes
char *p = (char *)&s;   // Force a char* pointer to look at the start of the short
```

If `s` has the value `514` (which is `0x0202` in Hex), its physical memory contains two separate bytes: `0x02` and `0x02`.
When we read `*p`, we are reading only **the first byte** (`0x02`, which is `2` in decimal), because `p` is a pointer to a `char` (1 byte).

<Callout type="pro">
**Type Punning**
This technique of reading a type's memory through a pointer of another type is called "Type Punning". It is powerful for systems programming, but it can cause undefined behavior if strict aliasing and alignment rules are not respected.

<Callout type="danger" title="Strict Aliasing Rule">
  The compiler assumes that two pointers of different types (e.g., `int*` and `float*`) never point to the same memory area. If you break this rule with casting, the optimizer might delete parts of your code!
  
  **Example**: If you change an `int` via a `char*`, the compiler might think the `int` never changed and use the old value saved in a **CPU Register** instead of re-reading from RAM.
</Callout>
</Callout>

### Endianness: The Byte Wars

**Endianness** defines the order in which the bytes composing a multi-byte number (like a 2-byte `short` or a 4-byte `int`) are stored in physical memory.

- **Little Endian** (Intel/AMD x86 architectures): The _Least Significant Byte_ (LSB) is stored at the lowest memory address.
- **Big Endian** (TCP/IP Network protocols, older Motorola CPUs): The _Most Significant Byte_ (MSB) is stored at the lowest address.

<Callout type="info" title="Why Little Endian?">
  Even though Big Endian feels more intuitive to humans (reading left-to-right
  like a decimal number), **Little Endian** is often preferred by processors
  because it allows treating a 32-bit number as an 8-bit one by simply reading
  the first byte, without needing to calculate offsets to the end of the data.
  It's a "free truncation" at the hardware level!
</Callout>

<Callout type="pro" title="Network Byte Order">
  The Internet speaks **Big Endian**. Since every computer has its own
  Endianness, when you send a number over the network you must use functions
  like `htons()` (Host TO Network Short) to ensure the recipient doesn't receive
  the bytes in reverse.
</Callout>

### Pointer Swapping: Pure Efficiency

Salvatore shows us a fundamental trick for simulations (like Game of Life): instead of copying thousands of bytes from the "new grid" to the "old grid" every turn, we simply swap the two **memory addresses**.

```c
char *temp = old_ptr;
old_ptr = new_ptr;
new_ptr = temp;
```

It's an instantaneous operation regardless of data size!

Example using the number `0x11223344`:

- In **Little Endian** memory: `44 33 22 11` (base address holds `44`).
- In **Big Endian** memory: `11 22 33 44` (base address holds `11`).

## C Strings vs Length-Prefixed Strings

Until now, we've used classic C strings (known as _Null-terminated strings_). They end with the `0x00` byte.

However, these strings have geometric problems:

1. To find the length, you must scan the entire string (Complexity `O(N)`).
2. They cannot contain `0x00` inside them, because it would be interpreted as the end!

A historical alternative (made famous by the Pascal language) is the **Length-Prefixed String**.
In this format, the **first byte** stores the total length of the string.

```c
// Pascal String. The first byte (11) dictates the length.
char *pascal_str = "\x0BHello World"; // \x0B is 11 in hex.
```

This makes getting the length an instantaneous operation (`O(1)`), simply by reading `pascal_str[0]`. The downside of using a single byte? A maximum length of 255 characters!

// Pascal-style string: first char = 5 (length), then "Hello"

    // Read the length:
    int len = str[0];
