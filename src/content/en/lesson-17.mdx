### xdump: Seeing the Memory

Instead of writing complex code, we use the concept of `xdump` to visualize bytes. In C, this means iterating over an `unsigned char*` pointer.

<Callout type="pro" title="Why unsigned char?">
  We use `unsigned char` for memory dumps because we want to see the **actual value** of the byte (0-255) without the sign bit (used in standard `char`) messing up our calculations or visualization.
</Callout>

## Memset: Deep Cleaning

In C, newly allocated memory or memory declared on the stack contains "garbage". With `memset`, we can zero everything out in one go.

<BinaryVisualizer 
  title="Effect of memset(p, 0, 4)"
  description="Before and after clearing an integer in memory."
  data={[
    { label: "Before (Garbage)", bytes: ["A2", "3F", "00", "B1"], address: "0x300" },
    { label: "After memset", bytes: ["00", "00", "00", "00"], address: "0x300" }
  ]}
/>

<Infographic 
  title="How memset Works"
  description="Sets every byte of a memory block to a specific value (usually 0)."
  items={[
    { label: "Pointer", value: "The starting address of the block", color: "blue" },
    { label: "Value", value: "The byte to copy (e.g., 0)", color: "emerald" },
    { label: "Size", value: "How many total bytes to flatten", color: "amber" }
  ]}
/>

<Callout type="info" title="memset Optimization">
  The compiler is so smart that if it sees a `memset` immediately followed by a `free`, it might decide **not to execute it** (because it "serves no purpose" for the logic of the program). This is a problem if you are zeroing out a password in memory for security! In those cases, special functions like `explicit_bzero` are used.
</Callout>

### Final Summary
We have learned not to trust abstractions: memory is a tape of bytes and we now have the tools to inspect and manipulate it with surgical precision. `memset` and `xdump` will be your best friends in low-level debugging.
