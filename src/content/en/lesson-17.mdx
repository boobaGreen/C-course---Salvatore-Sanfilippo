### xdump: Seeing the Memory

Instead of writing complex code, we use the concept of `xdump` to visualize bytes. In C, this means iterating over an `unsigned char*` pointer.

<Callout type="pro" title="Why unsigned char?">
  We use `unsigned char` for memory dumps because we want to see the **actual
  value** of the byte (0-255). If we used a simple `char`, bytes with the high
  bit set to 1 would be interpreted as negative numbers (Sign Extension) due to
  Two's Complement representation, messing up our hexadecimal visualization.
</Callout>

## Memset: Deep Cleaning and Security Risks

In C, newly allocated memory or memory declared on the stack contains "garbage" (data left over from previous programs). With `memset`, we can zero everything out in one go.

<BinaryVisualizer
  title="Effect of memset(p, 0, 4)"
  description="Before and after clearing an integer in memory."
  data={[
    {
      label: "Before (Garbage)",
      bytes: ["A2", "3F", "00", "B1"],
      address: "0x300",
    },
    {
      label: "After memset",
      bytes: ["00", "00", "00", "00"],
      address: "0x300",
    },
  ]}
/>

<Callout type="danger" title="Optimization That Kills Security">
  Compilers (GCC/Clang) always try to eliminate "useless" code. If you write a
  password to a buffer, use it, call `memset` to zero it out, and then the
  buffer goes out of scope, the compiler might **remove the memset** because "no
  one will ever read that buffer again." This leaves the password in plain text
  in RAM! **Solution**: In professional C, used `explicit_bzero()` or
  `memset_s()`, which the compiler is not allowed to optimize away.
</Callout>
