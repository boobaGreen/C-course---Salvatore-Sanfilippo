## File Descriptors: 0, 1, 2

Every Linux process is born with three already open communication channels:
- **0**: Standard Input (keyboard)
- **1**: Standard Output (terminal)
- **2**: Standard Error (errors on the terminal)

<Callout type="info" title="Why numbers?">
The Kernel is lazy (or rather, efficient). It doesn't want to handle long strings for every file. It uses a simple array for each process, and the index of that array is our **File Descriptor**.
</Callout>

## Example: Low-Level Writing

We use `open` and `write` instead of the standard functions. Note how we must handle octal permissions (e.g., 0644).

<CodeEditor 
  initialCode={`#include <stdio.h>
#include <unistd.h>
#include <fcntl.h>
#include <errno.h>
#include <string.h>

int main() {
    // Open a file in write-only mode, creating it if missing
    int fd = open("low_level.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);
    
    if (fd == -1) {
        perror("Error opening");
        return 1;
    }

    char *msg = "Greetings from the Kernel!\\n";
    if (write(fd, msg, strlen(msg)) == -1) {
        perror("Error writing");
    }

    close(fd);
    return 0;
}`} 
  language="c" 
/>

<Callout type="hacker" title="Ancient but Current">
The system calls you see here (like `open`) were defined in the 1970s for Unix. Fifty years later, they still power Linux, macOS, and almost the entire global cloud. Learning them means connecting with the "pioneers" of computing.
</Callout>

### errno and perror
When a syscall fails (returns -1), the Kernel sets a global number called `errno`. The `perror` function reads that number and translates it into a human-readable message (e.g., "No such file or directory").

### Final Summary
We have broken through the wall of the standard library. You now know that every time you write something to the screen or disk, you are asking for the Kernel's permission. Mastering syscalls means having total control over the machine.
