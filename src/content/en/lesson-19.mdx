## File Descriptors: 0, 1, 2

Every Linux process is born with three communication channels already open (inherited from the parent process):

- **0**: Standard Input (`stdin`) - Keyboard
- **1**: Standard Output (`stdout`) - Terminal
- **2**: Standard Error (`stderr`) - Terminal (for critical messages)

<Callout type="pro" title="Everything is a File">
  This is the Zen philosophy of Unix: files on disk, keyboards, monitors, and
  even network cards are all treated as **File Descriptors**. You can use
  `write()` to save a document or to send a message over the internet: for the C
  programmer, the logic is the same.
</Callout>

### The Architecture: Ring 3 vs Ring 0

The CPU does not trust your code. Your program runs in **User Mode (Ring 3)**, where it cannot directly touch the disk or other processes' memory. To do so, it must invoke a **System Call**.

Why not use `write` for everything?

## Low-Level Programming Exercise

<CodeEditor 
  initialCode={`#include <unistd.h> // Header for POSIX Syscalls
#include <string.h>
#include <errno.h>
#include <stdio.h>

int main() {
const char \*msg = "Direct writing to Terminal (fd 1)\\n";

    // write(file_descriptor, buffer, length)
    // No buffering: data is sent immediately!
    ssize_t written = write(1, msg, strlen(msg));

    if (written == -1) {
        // If write fails, errno is set by the Kernel
        perror("Error in write");
        return 1;
    }

    const char *err_msg = "This is an error (fd 2)\\n";
    write(2, err_msg, strlen(err_msg));

    return 0;

}`}
language="c"

### Final Summary

We have broken through the wall of the standard library. You now know that every time you write something to the screen or disk, you are asking for the Kernel's permission. Mastering syscalls means having total control over the machine and understanding how the Linux/Unix world really works.
